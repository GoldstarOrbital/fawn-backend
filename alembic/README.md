# Alembic migrations

Set up 2026-07-19 to replace the ad-hoc `ALTER TABLE` patching in
`main.py`'s `_init_db_schema()` (23 patches and counting, with no
rollback story and no protection against two instances racing to alter
the schema on the same deploy).

**Current state: production stamped (2026-07-19, revision `c8a93dfde825`)
via a one-time bootstrap endpoint that ran `alembic stamp head` inside
the app's own process. That endpoint has since been removed from the
codebase (see git history: `routers/admin_credit.py`) -- it was a
single-use bootstrap tool, not a permanent admin primitive. `main.py`
still self-migrates via the old `_patch()` system on every boot for now;
wiring `alembic upgrade head` into app startup (retiring `_patch()` for
new changes) is the next step, not done yet.**

The baseline migration (`alembic/versions/c8a93dfde825_*.py`) was
generated by diffing `models.py` against a genuinely empty database, so
it creates the *entire* schema from scratch. Verified before adopting:
applying it to a fresh empty SQLite DB via `alembic upgrade head`, then
running the full test suite against it, passed all 165 tests then
current -- it produces the same schema `Base.metadata.create_all()` does.

## Adding a new migration

```
alembic revision --autogenerate -m "add whatever_column to whatever_table"
# review the generated file in alembic/versions/ before applying --
# autogenerate is a good first draft, not a guarantee
alembic upgrade head
```
