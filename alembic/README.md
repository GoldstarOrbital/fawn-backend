# Alembic migrations

Set up 2026-07-19 to replace the ad-hoc `ALTER TABLE` patching in
`main.py`'s `_init_db_schema()` (23 patches and counting, with no
rollback story and no protection against two instances racing to alter
the schema on the same deploy).

**Current state: set up and verified locally. NOT yet wired into
production.** `main.py` still self-migrates via the old `_patch()`
system on every boot -- that keeps working exactly as before. Nothing
here changes production behavior until the one-time adoption step below
is deliberately run.

The baseline migration (`alembic/versions/c8a93dfde825_*.py`) was
generated by diffing `models.py` against a genuinely empty database, so
it creates the *entire* schema from scratch. Verified: applying it to a
fresh empty SQLite DB via `alembic upgrade head`, then running the full
test suite against it, passes all 165 tests -- it produces the same
schema `Base.metadata.create_all()` does today.

## Adopting this for production (one-time, not yet done)

Production's database already has this exact schema (built up over time
via `create_all()` + the `_patch()` calls) -- so the one-time adoption
step is `alembic stamp head`, which marks the baseline as already-applied
**without running any DDL against the live database**. It does not
create, alter, or drop anything; it just writes one row to a new
`alembic_version` tracking table so Alembic knows where the DB already
stands.

```
# Point DATABASE_URL at production, then, from a safe environment:
alembic stamp head
```

Do this deliberately, not as part of an automated deploy step, until
there's been at least one successful production stamp + a following
real migration to prove the round trip. After that, wiring
`alembic upgrade head` into the app's startup sequence (replacing the
`_patch()` calls in `main.py`) is the natural next step -- not done yet.

## Adding a new migration (once adopted)

```
alembic revision --autogenerate -m "add whatever_column to whatever_table"
# review the generated file in alembic/versions/ before applying --
# autogenerate is a good first draft, not a guarantee
alembic upgrade head
```
